## v2.0 Bug Report — Complete Analysis

I investigated the extension thoroughly via live console logs, network request monitoring, and direct API testing. Here are all confirmed bugs and exact fixes needed.

---

### Bug 1 — 404 Flood on Initial Load (`content.js`)

**Root cause:** The `MutationObserver` triggers when the grid's shadow DOM first appears, but the rows have skeleton/placeholder IDs (`he-row-0`, `1`, `2`, ... `9`) not real UUIDs. The extension injects cells for those bogus IDs AND fires single-customer fetch requests for each (`/delegatedAdminGdapCustomers?$filter=tenantId eq 'he-row-0'`). All 10 return 404.

**Console evidence:**
```
[AltName] Row cell injected: he-row-0 -> Unknown
[AltName] Row cell injected: 1 -> Unknown
[AltName] Single-customer fetch failed for he-row-0 status 404
[AltName] Single-customer fetch failed for 1 status 404
```

**Fix:** Before injecting a cell or firing any single-customer fetch, validate the row ID is a proper UUID v4 (matches regex `/^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i`). Skip any row whose ID doesn't match. The real rows arrive ~700ms later via a second MutationObserver event when the actual data replaces the skeletons.

---

### Bug 2 — `Something went wrong` Banner / Search Returns No Results

**Root cause: OData node count limit of 30**

The API enforces a hard limit: `"The node count limit of '30' has been exceeded"` (confirmed via actual API response body). The current filter building creates **nested parentheses** which consume far more AST nodes than a flat OR chain:

**Current (broken) nested format:**
```
((((contains(tolower(displayName), tolower('demo')) OR contains(tolower(tenantId), tolower('demo'))) OR tenantId eq 'a') OR tenantId eq 'b') OR tenantId eq 'c') OR tenantId eq 'd'
```
Result: 4+ clauses → HTTP 400. **Maximum allowed with this format: 3 clauses.**

**Flat format (better):**
```
(contains(tolower(displayName), tolower('demo')) OR contains(tolower(tenantId), tolower('demo')) OR tenantId eq 'a' OR tenantId eq 'b' OR tenantId eq 'c' OR tenantId eq 'd')
```
This uses fewer nodes. UUID-only flat OR allows up to 6 clauses before hitting the limit. Combined with the base `contains()` filter (which uses ~12 nodes), a flat format should allow approximately 4–5 `tenantId eq` clauses.

**Fix:** Build the expanded filter as a single flat OR expression. Specifically, instead of wrapping the original filter in successive `OR tenantId eq` additions, extract the base filter's inner content and add the tenant IDs alongside it in one flat group:

```javascript
// CURRENT (broken):
let newFilter = origFilter;
for (const id of matchingIds) {
  newFilter = `(${newFilter} OR tenantId eq '${id}')`;
}

// FIXED (flat):
// Strip outer parens from origFilter, then build flat:
const inner = origFilter.replace(/^\(|\)$/g, '');
const tenantClauses = matchingIds.map(id => `tenantId eq '${id}'`).join(' OR ');
const newFilter = `(${inner} OR ${tenantClauses})`;
```

**Additionally, add a hard cap:** If `matchingIds.length > 4` (to be safe within the ~30 node budget with flat format), skip the expansion entirely rather than generating a 400. The user sees the displayName-only results, which is better than a red error banner.

### Bug 3 — Over-Matching: `p:` Domain Names Included in AltName Search

**Root cause:** The extension stores and matches **both** `p:` (partner domain, auto-populated) and `c:` (user custom name) entries in the altNames map sent to `search-inject.js`. When the user types "a", there are 36 altName matches because nearly every customer's domain contains "a". This generates massive filters that always exceed the node limit.

**Console evidence:**
```
[AltName/net] Expanded $filter for query "a" + 36 alt-name match(es).
[AltName/net] Expanded $filter for query "d" + 30 alt-name match(es).
```

**Fix:** In `content.js`, when building the altNames object to send via `postMessage` to `search-inject.js`, **only include entries with a `c:` prefix** (user-set custom names). Strip `p:` entries entirely from the map used for search expansion. The `p:` data can still be displayed in the Alternative Name column, but should not drive search expansion.

```javascript
// When posting altNames to search-inject.js, filter to c: only:
const searchableAltNames = {};
for (const [tenantId, value] of Object.entries(allAltNames)) {
  if (value.startsWith('c:')) {
    searchableAltNames[tenantId] = value.slice(2); // strip 'c:' prefix
  }
}
window.postMessage({ type: 'ALTNAMES_UPDATE', altNames: searchableAltNames }, '*');
```

### Bug 4 — Ghost/Stale AltName Entry Causes Valid 3-Name Search to Fail

**Root cause:** The "demo" search filter contained 4 `tenantId eq` clauses (→ 400), even though the user only set 3 custom names ("demo X", "demo Y", "some other demo"). The 4th entry (`a1b2c3d4-0005-4abc-8def-111122227777`) is a stale or `p:`-domain entry that matched "demo" in the domain name. After the fix in Bug 3 (exclude `p:` entries from search), this 4th ghost match will be eliminated automatically.
