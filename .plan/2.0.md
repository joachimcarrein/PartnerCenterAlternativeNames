## Prompt for Development Agent

You are working on a Chrome Extension that enhances the Microsoft Partner Center **Customers | Granular Administration** page (`https://partner.microsoft.com/dashboard/v2/customers/granularadminaccess/list`).

The extension currently adds an **"Alternative Name"** column to the customer list table. Your task is to implement **alternative name search integration** so that when a user types in the search box, it also matches against alternative names stored by the extension.

---

### Current Extension Behaviour

The extension injects directly into the **Shadow DOM** of `customersvcadmin_he-data-grid`. For each customer row it:

1. Adds a `<th id="tenant-domain-header">Alternative Name</th>` column header
2. Adds `<td id="cell-tenantDomain-{tenantId}" class="data-grid__cell">` cells containing:
   - `<span class="altname-value">` — displays the name
   - `<button class="altname-edit">✎</button>` — opens an inline edit input
   - `<button class="altname-reset">↺</button>` — resets to default (only shown for custom names)
3. The wrapping `<span class="data-grid__cell-content-wrapper">` has a `data-rendered` attribute encoding the value source:
   - `"p:contoso.onmicrosoft.com"` → `p:` prefix = default partner domain (fetched from a separate source, not the main list API)
   - `"c:Acme Corp — Demo Client"` → `c:` prefix = custom name set by the user

Custom names (`c:` prefix values) are persisted in **`chrome.storage.local`**.

### How the Default Search Works (Full Technical Detail)

**The search is 100% server-side.** Here is the exact flow:

1. The search box is a custom Web Component: `<customersvcadmin_he-search-box debounce="20">`. Its Shadow DOM contains a native `<input type="search">`.

2. On each keystroke, the component fires `he-input` and `input` events (bubbling) with the current value on `e.target.value`.

3. A **React component** (located 8 fiber levels above the search box element in the React tree) listens for this and updates its own state. It builds an **OData `$filter` string**:
   ```
   (contains(tolower(displayName), tolower('QUERY')) OR contains(tolower(tenantId), tolower('QUERY')))
   ```

4. This filter is stored in **React state hook index 6** at that fiber depth.

5. The React component makes a **GET XHR request** to:
   - Host: `api.partnercustomersecurity.microsoft.com`
   - Query parameters: `$filter`, `$orderby`, `$count=true`

6. The API returns rows where each row has **only two data fields**: `displayName` and `tenantId`. The Alternative Name / tenant domain is **never part of the API response** — it is always client-side only.

**The consequence:** Searching "Acme Corp — Demo Client" returns zero results because the customer's `displayName` is "Acme Corporation" and the API has no knowledge of the custom name "Acme Corp — Demo Client".

### The Implementation to Build

Use **Approach C: React State Dispatch Interception**, augmented with a client-side post-filter fallback for results that don't appear in the API response.

#### How to Access the React Filter Dispatch

The filter `setState` dispatch function is accessible at runtime via the React fiber tree:

```javascript
function getFilterDispatch(searchBoxElement) {
  // Get the React fiber node attached to the search box element
  const fiberKey = Object.keys(searchBoxElement).find(k => k.startsWith('__reactFiber'));
  if (!fiberKey) return null;
  
  let fiber = searchBoxElement[fiberKey];
  
  // Walk up exactly 8 levels to reach the component that owns the filter state
  for (let i = 0; i < 8; i++) {
    fiber = fiber.return;
    if (!fiber) return null;
  }
  
  // Navigate the memoizedState linked list to index 6 (the OData filter state)
  let stateNode = fiber.memoizedState;
  for (let i = 0; i < 6; i++) {
    stateNode = stateNode?.next;
    if (!stateNode) return null;
  }
  
  return stateNode?.queue?.dispatch; // This is the React setState function for the filter
}
```

You can verify you have the right state node because `stateNode.memoizedState` will be a string matching the pattern `(contains(tolower(displayName), tolower('...'))...)`.

#### The Search Enhancement Logic

```javascript
async function enhanceSearch(query) {
  if (!query || query.trim() === '') return; // Let React handle empty/clear naturally

  const trimmedQuery = query.trim().toLowerCase();

  // 1. Load all stored alternative names from chrome.storage.local
  //    Expected storage schema: { altNames: { [tenantId]: string } }
  const data = await chrome.storage.local.get('altNames');
  const altNames = data.altNames || {}; // { tenantId: "Custom Name" }

  // 2. Also include default "p:" values currently rendered in the DOM
  //    (in case the extension has tenant domains cached in the DOM)
  const domAltNames = {};
  const grid = document.querySelector('customersvcadmin_he-data-grid');
  if (grid && grid.shadowRoot) {
    grid.shadowRoot.querySelectorAll('[id^="cell-tenantDomain-"]').forEach(cell => {
      const wrapper = cell.querySelector('[data-rendered]');
      if (wrapper) {
        const raw = wrapper.getAttribute('data-rendered') || '';
        const value = raw.startsWith('c:') || raw.startsWith('p:') ? raw.slice(2) : raw;
        const tenantId = cell.id.replace('cell-tenantDomain-', '');
        domAltNames[tenantId] = value;
      }
    });
  }

  // Merge: chrome.storage custom names take precedence over DOM-observed defaults
  const allAltNames = { ...domAltNames, ...altNames };

  // 3. Find tenantIds where the alternative name contains the query
  const matchingTenantIds = Object.entries(allAltNames)
    .filter(([, name]) => name.toLowerCase().includes(trimmedQuery))
    .map(([tenantId]) => tenantId);

  if (matchingTenantIds.length === 0) {
    // No alt name matches — let the original API filter do its work unchanged
    return;
  }

  // 4. Build the expanded OData filter
  //    Keep the original filter clauses AND add OR tenantId eq '...' for each match
  const altNameClauses = matchingTenantIds
    .map(id => `tenantId eq '${id}'`)
    .join(' OR ');

  const expandedFilter =
    `(contains(tolower(displayName), tolower('${query}')) ` +
    `OR contains(tolower(tenantId), tolower('${query}')) ` +
    `OR ${altNameClauses})`;

  // 5. Inject the expanded filter into React state via the dispatch function
  const searchBox = document.querySelector('customersvcadmin_he-search-box');
  if (!searchBox) return;

  const dispatch = getFilterDispatch(searchBox);
  if (typeof dispatch === 'function') {
    dispatch(expandedFilter);
  }
}
```

#### Event Listener Registration

Attach the listener in **capture phase** so it fires before React processes the event. Add a small delay (30ms, slightly more than the component's 20ms debounce) to ensure React has already set its own filter before we override it:

```javascript
function attachSearchEnhancement() {
  const searchBox = document.querySelector('customersvcadmin_he-search-box');
  if (!searchBox) return;

  searchBox.addEventListener('he-input', (e) => {
    const query = e.target.value;
    // Wait 30ms (debounce is 20ms) to let React fire its own setState first,
    // then override with our expanded filter
    setTimeout(() => enhanceSearch(query), 30);
  }, true); // capture phase
}
```

#### Storage Schema

Ensure the existing alt-name save/edit logic writes to `chrome.storage.local` in this exact schema:

```javascript
// When saving a custom name:
async function saveAltName(tenantId, customName) {
  const data = await chrome.storage.local.get('altNames');
  const altNames = data.altNames || {};
  altNames[tenantId] = customName;
  await chrome.storage.local.set({ altNames });
}

// When resetting to default:
async function resetAltName(tenantId) {
  const data = await chrome.storage.local.get('altNames');
  const altNames = data.altNames || {};
  delete altNames[tenantId];
  await chrome.storage.local.set({ altNames });
}
```

If the extension currently uses a different storage key or schema, adapt the `enhanceSearch` function accordingly.

### Important Technical Constraints

1. **React fiber depth is fragile.** The `8` level walk-up and state index `6` were confirmed by live inspection. If the page is updated and the search stops working, the depth or index may need re-checking. Consider adding a validation guard: confirm `stateNode.memoizedState` is a string containing `'contains(tolower(displayName)'` before calling dispatch.

2. **The fiber key prefix is `__reactFiber`** (confirmed on this page). Some React versions use `_reactFiber` — the `find()` call above handles both.

3. **Shadow DOM access:** The data-grid element is `customersvcadmin_he-data-grid` and its shadow root is always accessible via `grid.shadowRoot` on this page (no closed shadow roots).

4. **The `he-input` event** is the correct event to listen for (not `input` or `change`). It fires on every keystroke with the full current value on `e.target.value`.

5. **Page load timing:** The `customersvcadmin_he-data-grid` and `customersvcadmin_he-search-box` elements are rendered dynamically. Use a `MutationObserver` on `document.body` to detect when these elements appear before calling `attachSearchEnhancement()`.

6. **OData filter syntax:** The API uses OData v4. The `tenantId eq '...'` clause is safe to append. Do not use `contains()` on tenantId for the alt-name expansions — use exact `eq` matching since you have the exact tenantId.

7. **Clearing search:** When the query is empty (user clears the search box), the dispatch will be called with an empty string or the original unconstrained filter. The extension does not need to do anything special for this case — React handles it naturally.
